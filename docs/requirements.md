# Requirements Document: Food Delivery Platform

## 1. Функциональные требования (ФТ)

ФТ-001: Пользователь может искать рестораны по названию, кухне, рейтингу и расстоянию.

ФТ-002: Пользователь может просматривать меню ресторана, включая блюда, цены, описание и доступность.

ФТ-003: Пользователь может добавлять блюда в корзину и редактировать состав заказа.

ФТ-004: Пользователь может оформить заказ, указав адрес доставки, комментарий и время доставки.

ФТ-005: Пользователь может отслеживать статус доставки заказа в реальном времени.

---

# 2. Нефункциональные требования (НФТ)

## НФТ-001: Задержка (Latency)

SLI (что измеряем): p99 задержка API ключевых пользовательских операций

SLO (целевое значение):

* Поиск ресторанов: p99 < 200ms
* Просмотр меню: p99 < 300ms
* Добавление в корзину: p99 < 200ms
* Оформление заказа: p99 < 800ms
* Обновление статуса доставки: p99 < 300ms

Оценка архитектурной значимости:

* Влияние на структуру (Impact): High — требуется кэширование, CDN, оптимизация БД
* Стоимость изменения (Cost of Change): High — влияет на архитектуру сервисов
* Бизнес-риск (Business Risk): Medium — пользователи уходят если сайт работает медленно

Архитектурно-значимый: Да

---

## НФТ-002: Пропускная способность (Throughput)

SLI (что измеряем): количество запросов в секунду (RPS)

SLO:

* Средний RPS: ~250
* Пиковый RPS: до 5000
* Пиковый RPS обновления статусов: до 10000

Оценка архитектурной значимости:

* Impact: High — требуется горизонтальное масштабирование
* Cost of Change: Medium — можно масштабировать инфраструктуру
* Business Risk: Medium — деградация UX при перегрузке

Архитектурно-значимый: Нет

---

## НФТ-003: Доступность (Availability)

SLI: процент успешных запросов

SLO:

* Оплата: 99.99%
* Оформление заказа: 99.95%
* Каталог ресторанов: 99.9%
* Уведомления: 99.5%

Оценка архитектурной значимости:

* Impact: High — требуется репликация и отказоустойчивость
* Cost of Change: High — мультизонная архитектура
* Business Risk: High — простой системы приводит к потере заказов

Архитектурно-значимый: Да

---

## НФТ-004: Сохранность данных (Durability)

SLI: вероятность потери данных

SLO:

* Платежные транзакции: 99.999999999%
* Заказы: 99.999999%
* История заказов: 99.999%

Оценка архитектурной значимости:

* Impact: High — требуется надёжное хранение и репликация
* Cost of Change: High — изменение storage критично
* Business Risk: High — потеря платежей недопустима

Архитектурно-значимый: Да

---

## НФТ-005: Масштабируемость (Scalability)

SLI: способность системы обрабатывать рост нагрузки

SLO:

* MAU до 50 млн пользователей
* до 2 млн заказов в день
* до 20k RPS
* до 200k онлайн курьеров

Оценка архитектурной значимости:

* Impact: High — требуется горизонтальная архитектура
* Cost of Change: High — монолит сложно масштабировать
* Business Risk: Medium — рост бизнеса невозможен без масштабирования

Архитектурно-значимый: Да

# 3. Расчёт нагрузки (Capacity Estimation)

## 3.1 Пользователи

```
MAU: 20 000 000

DAU: 2 000 000
(20M × 0.10, предположение: 10% активных пользователей)
```

Пиковый сценарий: обеденный пик (12:00–14:00), ужин (18:00-20:00)
пиковый коэффициент: x3 (предположение) 

---

## 3.2 RPS

Соотношение чтение/запись: 90/10

Предположение:

```
Req/user/day ≈ 10
```

Средний RPS:

```
Средний RPS = DAU × Req/user / 86400
            = 2 000 000 × 10 / 86400
            ≈ 230 RPS
```

Peak RPS:

```
Peak RPS = (DAU × Req/user × Peak_coef) / (Peak_hours × 3600)
         = (2 000 000 × 10 × 3) / (4 × 3600)
         ≈ 4166 RPS
```

---

## 3.3 Бюджет задержки (Latency Budget)

Endpoint: POST /api/order
SLO: p99 < 800ms

| Компонент            | Задержка (мс) | Комментарий        |
| -------------------- | ------------- | ------------------ |
| API Gateway          | 20            | балансировщик      |
| Order Service        | 80            | бизнес логика      |
| Cart Service         | 50            | чтение корзины     |
| Payment Service      | 200           | подготовка платежа |
| DB write             | 40            | запись заказа      |
| Payment provider API | 300           | внешний сервис     |
| Network              | 50            | сеть               |
| ИТОГО                | 740           |                    |
| Буфер (30%)          | 222           |                    |
| С БУФЕРОМ            | 962           |                    |
| Остаток от SLO       | -162          | ❌                  |

Узкое место (Bottleneck): внешний платёжный провайдер.

Предлагаемое решение:

* асинхронное подтверждение оплаты
* retry механизмы
* idempotent payment API

## 3.4 Хранилище (Storage)

Формула:

```
Storage = записей/день × размер_записи × дней × коэффициент_репликации
```

Расчёт заказов:

```
400K заказов/день
Размер записи ≈ 1 KB

400K × 1KB = 400 MB/день
400MB × 365 = 146 GB/год

репликация ×3

≈ 438 GB/год
```

| Тип данных | Объём через 1 год | Объём через 3 года | Комментарий         |
| ---------- | ----------------- | ------------------ | ------------------- |
| Заказы     | ~438 GB           | ~1.3 TB            | с репликацией       |
| Фото меню  | ~500 GB           | ~600 GB            | статический контент |
| Логи       | ~3 TB             | ~9 TB              | хранение 30 дней    |

---

## 3.5 Bandwidth

Формулы:

```
Входящий  = Peak_RPS × avg_request_size
Исходящий = Peak_RPS × avg_response_size
```

Предположения:

```
avg_request_size = 2 KB
avg_response_size = 10 KB
Peak_RPS = 5000
```

Расчёт:

```
Входящий:  5000 × 2KB  = 10 MB/s  ≈ 80 Mbps
Исходящий: 5000 × 10KB = 50 MB/s  ≈ 400 Mbps
```

# 4. Допущения, ограничения и открытые вопросы

## Допущения

1. DAU составляет около 10% от MAU.
2. Пиковые часы в обед и ужин, коэффициент 3
3. Средний пользователь делает один заказ в неделю.
4. Средний размер API ответа ~10KB.
5. Размер записи заказа ~1KB.

---

## Ограничения

1. Ограниченный бюджет инфраструктуры — используются managed cloud сервисы.
2. География: Россия и СНГ — возможны сетевые задержки между регионами.
3. Нагрузка сильно неравномерна (пики в обеденное и вечернее время).

---

## Открытые вопросы

1. Сколько ресторанов планируется подключить к системе?
2. Используется ли собственная курьерская сеть или сторонние сервисы доставки?
3. Нужно ли отображение курьера на карте в реальном времени?
4. Какие методы оплаты обязательны (карта, СБП, наличные)?
5. Требуется ли прогнозирование времени доставки?

---

## Выявленные противоречия

1. Требование высокой производительности при ограниченном бюджете
   → Решение: использование CDN, кэширования и object storage.

2. Требование мгновенного оформления заказа и одновременно надёжной оплаты
   → Решение: асинхронное подтверждение оплаты и идемпотентность транзакций.

3. Требование real-time статусов доставки при нестабильной мобильной сети
   → Решение: event-driven обновления + fallback polling.
